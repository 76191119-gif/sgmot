ACTIVIDAD 1 — Evaluación de Calidad del Proyecto (ISO 25010)

Modalidad: Grupal  |  Duración estimada: 90 minutos

1.1 Descripción General del Sistema

SGMOT es una aplicación web orientada a operadores de servicios de telecomunicaciones (ISP) que centraliza la gestión de clientes, técnicos de campo, órdenes de trabajo e incidencias técnicas, junto con un sistema de notificaciones en tiempo real y trazabilidad de auditoría. Su arquitectura está compuesta por un backend en PHP puro con PDO y JWT, y un frontend reactivo construido con React 18, Vite y TailwindCSS, comunicados a través de una API REST. La base de datos es MySQL 8 con cinco entidades relacionales principales: users, clients, technicians, work_orders, incidents, complementadas por tablas de notifications y audit_logs.

1.4 Plan de Acciones Prioritarias para Cerrar Brechas

Acción Prioritaria 1 — Implementar Suite de Pruebas Unitarias e Integración

Objetivo: Elevar la cobertura de pruebas desde 0 % hasta ≥ 60 % en las rutas críticas del backend y los componentes clave del frontend.

Justificación: La ausencia total de pruebas automatizadas representa la brecha de mayor riesgo para la mantenibilidad y fiabilidad del sistema. Ante cualquier refactorización o adición de funcionalidad, el equipo depende exclusivamente de pruebas manuales, lo cual es ineficiente y propenso a errores.

Actividades concretas:

  • Backend: integrar PHPUnit 10. Priorizar pruebas sobre auth.php, register.php y work_orders.php (rutas con mayor CC y riesgo de defecto).
  • Frontend: integrar Vitest + React Testing Library. Priorizar pruebas sobre AuthContext, usePermissions y los formularios de registro y login.
  • Plazo estimado: 3 semanas (sprint dedicado).

Acción Prioritaria 2 — Refactorización y Robustecimiento de Manejo de Errores en el Backend

Objetivo: Eliminar las brechas de fiabilidad y mantenibilidad corrigiendo los 3 defectos críticos identificados en la revisión de código (Actividad 2).

Actividades concretas:

  • Envolver el bloque POST de work_orders.php (línea 99) en un try-catch PDOException, detectar SQLSTATE 23000 (violación de clave única) y regenerar el número de orden con un identificador robusto (UUID v4 o timestamp en microsegundos).
  • Agregar validación de campos obligatorios en generateToken($user) para prevenir tokens con campos null.
  • Refactorizar register.php extrayendo la lógica de CASO A y CASO B en funciones privadas independientes para reducir la CC de 13 a < 7.
  • Plazo estimado: 1 semana.

Acción Prioritaria 3 — Implementar Pruebas de Carga y Encuesta de Satisfacción de Usuario

Objetivo: Medir con datos reales las brechas de eficiencia de desempeño y usabilidad, estableciendo una línea base cuantitativa para futuros sprints de mejora.

Actividades concretas:

  • Ejecutar escenarios de carga con Apache JMeter o k6 simulando 50 usuarios concurrentes realizando login y consulta de órdenes. Documentar tiempos de respuesta P50, P95 y P99.
  • Diseñar y aplicar encuesta de satisfacción (SUS — System Usability Scale, 10 ítems) a un grupo piloto de 5 usuarios reales (1 admin, 2 técnicos, 2 clientes).

2.3 Función 1: generateToken($user)

Código fuente revisado

// Archivo: api/helpers/auth.php — Líneas 15–26
function generateToken($user) {
    $header  = b64url_encode(json_encode(['typ' => 'JWT', 'alg' => 'HS256']));
    $payload = b64url_encode(json_encode([
        'id'        => $user['id'],
        'email'     => $user['email'],
        'role'      => $user['role'],
        'full_name' => $user['full_name'],
        'exp'       => time() + 86400, // 24h
    ]));
    $sig = b64url_encode(hash_hmac('sha256', "$header.$payload", JWT_SECRET, true));
    return "$header.$payload.$sig";
}

¿Tiene comentarios para secciones complejas? ⚠ PARCIAL Solo existe el comentario inline // 24h para la expiración. La estructura JWT (header.payload.signature) y el algoritmo HS256 no están documentados para un lector no familiarizado.

2.5 Función 3: canAccessWorkOrder($db, $user, $order)

Código fuente revisado

// Archivo: api/routes/work_orders.php — Líneas 18–34
function canAccessWorkOrder($db, $user, $order) {
    if ($user['role'] === 'admin') return true;

    if ($user['role'] === 'tecnico') {
        $tech = getCurrentTechnician($db, $user);
        return $tech
            && (int)$order['technician_id'] === (int)$tech['id'];
    }

    if ($user['role'] === 'cliente') {
        $client = getCurrentClient($db, $user);
        return $client
            && (int)$order['client_id'] === (int)$client['id'];
    }

    return false;
}

Evaluación con Checklist

CriterioEstadoJustificación

¿Tiene comentarios para secciones complejas? ✘ NO La función carece de comentarios. Aunque la lógica es legible, una nota sobre el principio de acceso (rol admin = acceso irrestricto; otros roles = acceso por pertenencia) mejoraría la comprensión durante mantenimiento.

2.7 Los 3 Defectos Más Críticos Identificados

Rank Defecto Ubicación Severidad Descripción y Riesgo
1° Generación de número de orden no garantiza unicidad; ausencia de manejo de colisión work_orders.php línea 97 ALTA El número de orden se genera con rand(100, 999). Solo existen 900 valores posibles por día. Una colisión produce una excepción PDO no capturada (sin try-catch en el bloque POST), resultando en HTTP 500 con posible exposición de información interna y sin rollback transaccional.

ACTIVIDAD 3 — Diseño de Casos de Prueba desde Requisitos

Modalidad: Individual  |  Duración estimada: 45 minutos

3.1 Requisito Funcional Seleccionado

CampoDetalle
ID del Requisito RF-SEC-001
Nombre Autenticación segura de usuarios con bloqueo por intentos fallidos
Descripción El sistema debe autenticar a los usuarios mediante credenciales válidas (correo electrónico y contraseña). Debe retornar un token JWT válido al autenticar exitosamente y bloquear el acceso durante 15 minutos tras 5 intentos fallidos consecutivos provenientes de la misma dirección IP o correo electrónico.
Módulo Autenticación — api/routes/auth.php
Criterio de Calidad ISO 25010 Seguridad → Autenticidad: La capacidad del sistema para demostrar que la identidad de un sujeto o recurso es la que se declara ser.
Métrica que lo mide (M1) Tasa de éxito en autenticación con credenciales válidas: meta = 100 %
(M2) Tasa de bloqueo efectivo tras 5 intentos fallidos en 15 minutos: meta = 100 %
(M3) Tiempo de respuesta del endpoint POST /auth/login: meta < 500 ms
Criterio de Aceptación Condición de éxito: POST /auth/login con credenciales correctas retorna HTTP 200, cuerpo JSON con campo token (string no vacío) y objeto user con id, email, role y full_name correctos, en un tiempo inferior a 500 ms.

Condición de fallo (credenciales incorrectas): Retorna HTTP 401, campo error con mensaje "Credenciales incorrectas", sin campo token.

Condición de bloqueo (rate limiting): Tras 5 intentos fallidos del mismo email o IP en menos de 15 minutos, el 6.° intento (incluso con credenciales válidas) retorna HTTP 429, campo error con mensaje "Demasiados intentos fallidos. Intenta nuevamente en 15 minutos."

4.1 Registro del Defecto

CampoValor
ID del DefectoDEF-SGMOT-001
Título / Resumen Generación de número de orden con rand() no garantiza unicidad; colisión produce HTTP 500 no controlado sin rollback transaccional
Descripción detallada Cuando se crea una orden de trabajo mediante POST /api/work_orders sin incluir el campo order_number en el cuerpo de la solicitud, el sistema genera un identificador mediante la expresión 'OT-' . date('ymd') . '-' . rand(100, 999) (línea 97 de work_orders.php). Esta lógica produce únicamente 900 valores posibles por día de calendario. Si el número generado aleatoriamente coincide con el de una orden existente del mismo día, la base de datos lanza una excepción de violación de restricción UNIQUE (SQLSTATE 23000). Dado que el bloque POST no está envuelto en un bloque try-catch, la excepción PDO se propaga como error fatal de PHP, el servidor responde con HTTP 500 con información potencialmente sensible de la capa de datos, y no se ejecuta ningún rollback porque no hay transacción activa. La probabilidad de colisión aumenta geométricamente con el volumen de órdenes por día (problema del cumpleaños matemático).
Severidad ALTA — Interrumpe la operación de creación de órdenes, afecta la integridad del sistema y puede exponer información interna.
Prioridad ALTA — Debe corregirse antes del despliegue en producción; afecta el flujo de negocio principal del sistema.

4.2 Análisis de Causa Raíz — Método de los 5 Porqués

El método de los 5 Porqués es una técnica de análisis causal inductivo que consiste en preguntar repetidamente "¿por qué?" ante cada respuesta obtenida, hasta llegar a la causa raíz del problema. A continuación se aplica dicho método al defecto DEF-SGMOT-001.

# Pregunta Respuesta / Causa Identificada
¿Por qué 1? ¿Por qué el sistema devuelve HTTP 500 al crear una orden de trabajo? Porque se lanza una excepción PDOException (violación de restricción UNIQUE en el campo order_number) que no es capturada por ningún bloque try-catch en el bloque POST del archivo work_orders.php.
¿Por qué 2? ¿Por qué la excepción PDO no es capturada en el bloque POST? Porque el bloque POST de work_orders.php no fue implementado con manejo de errores explícito mediante try-catch. Solo el flujo de registro en register.php posee dicho manejo, pero no fue replicado de manera consistente en otras rutas.
¿Por qué 3? ¿Por qué no se implementó try-catch de forma consistente en todas las rutas? Porque no existe un estándar documentado ni un middleware centralizado de manejo de errores de base de datos en el proyecto. Cada ruta fue desarrollada de manera independiente, sin una guía de codificación que estableciera la obligatoriedad del manejo de excepciones PDO.
¿Por qué 4? ¿Por qué no existe un estándar de manejo de errores de BD en el proyecto? Porque se asumió que el número de orden generado con rand(100, 999) sería único con probabilidad suficiente. No se realizó un análisis cuantitativo del riesgo de colisión (problema del cumpleaños), lo que llevó a omitir el manejo del caso de duplicado como un escenario plausible.
¿Por qué 5? ¿Por qué se utilizó rand(100, 999) en lugar de un generador de IDs con garantía de unicidad? Causa raíz: Ausencia de una política de generación de identificadores únicos en los estándares de arquitectura del proyecto. No se evaluaron alternativas como UUID v4, secuencias de base de datos (AUTO_INCREMENT), o identificadores basados en timestamp de alta resolución + proceso ID. El desarrollador optó por una solución simple y rápida sin considerar el comportamiento del sistema bajo carga operativa real.

4.3 Acción Correctiva y Preventiva

Acción Correctiva (Inmediata) — Corrige el defecto existente

Reemplazar la generación de número de orden en la línea 97 de work_orders.php con un identificador basado en timestamp de alta resolución y número aleatorio de mayor entropía, garantizando la unicidad dentro de un contexto operativo razonable. Adicionalmente, envolver el bloque INSERT en un try-catch con detección específica de la violación UNIQUE (código de error MySQL 1062 / SQLSTATE 23000).

Código corregido propuesto:

// api/routes/work_orders.php — Bloque POST CORREGIDO
if ($method === 'POST') {
    // ... (validaciones previas sin cambio)

    $orderNum = $b['order_number']
        ?? ('OT-' . date('ymd') . '-' . strtoupper(substr(bin2hex(random_bytes(3)), 0, 6)));

    try {
        $stmt = $db->prepare("INSERT INTO work_orders
            (order_number, type, client_id, ...) VALUES (?, ?, ?, ...)");
        $stmt->execute([...]);
        $newId = (int)$db->lastInsertId();
    } catch (PDOException $e) {
        if ($e->getCode() === '23000') {
            sendResponse(['error' => 'Número de orden duplicado. Por favor intente nuevamente.'], 409);
        }
        sendResponse(['error' => 'Error al crear la orden. Intente nuevamente.'], 500);
    }

    // ... (auditoría + notificaciones sin cambio)
    sendResponse(['id' => $newId, 'order_number' => $orderNum, 'message' => 'Orden creada'], 201);
}

La función random_bytes(3) genera 3 bytes criptográficamente seguros que codificados en hexadecimal producen 6 caracteres (16³⁶ = 16,777,216 combinaciones posibles por día), reduciendo la probabilidad de colisión a un valor estadísticamente despreciable (p < 0.000006 % para 1,000 órdenes/día).

Acción Preventiva (Sistémica) — Evita que defectos similares ocurran en el futuro

Las siguientes acciones preventivas deben implementarse a nivel de estándar del proyecto:

  1. Estándar de codificación para rutas PHP: Establecer como obligatorio que todo INSERT/UPDATE que afecte restricciones de unicidad o claves foráneas esté envuelto en un try-catch PDOException con detección de SQLSTATE. Documentar en un archivo CONTRIBUTING.md dentro del repositorio.
  2. Checklist de code review: Agregar al proceso de revisión de código (ver Actividad 2) el ítem: "¿Las operaciones de escritura en BD manejan explícitamente las excepciones de integridad referencial y de unicidad?"
  3. Prueba unitaria de regresión: Una vez implementado PHPUnit (Acción Prioritaria 1 de la Actividad 1), crear un caso de prueba específico que simule la colisión de order_number y verifique que el sistema retorna HTTP 409 con mensaje amigable.
  4. Política de IDs: Establecer en la arquitectura del proyecto que todos los identificadores de entidades visibles externamente deben utilizar random_bytes() o UUID v4 (com_create_guid() / librería ramsey/uuid), reservando AUTO_INCREMENT para claves primarias internas no expuestas.

CONCLUSIONES GENERALES

El desarrollo de las cuatro actividades prácticas permitió aplicar de manera integral el modelo ISO/IEC 25010:2011 al sistema SGMOT, revelando tanto las fortalezas del proyecto como las brechas que deben ser abordadas antes de su despliegue en un entorno de producción real.

En términos de adecuación funcional, el sistema cubre de manera efectiva el 95 % de los requisitos operativos definidos para la gestión de órdenes de trabajo, incidencias técnicas y seguimiento de clientes en una empresa de telecomunicaciones. Las características de seguridad demuestran un nivel maduro: el uso de JWT firmado con HS256, contraseñas bcrypt, rate limiting y consultas preparadas reflejan buenas prácticas de desarrollo seguro, con una cobertura del 100 % de endpoints protegidos y ausencia comprobada de inyección SQL.

Sin embargo, la mantenibilidad constituye la brecha más crítica identificada: la cobertura de pruebas unitarias es del 0 %, lo que expone al proyecto a regresiones no detectadas ante cualquier cambio futuro. Complementariamente, la complejidad ciclomática elevada en register.php (CC = 13) y la ausencia de manejo de excepciones robusto en el módulo de órdenes de trabajo representan riesgos de fiabilidad y mantenibilidad que deben ser abordados de inmediato.

El defecto DEF-SGMOT-001 identificado en la Actividad 4 —generación no idempotente de números de orden mediante rand(100, 999)— constituye el hallazgo de mayor impacto potencial, dado que su probabilidad de manifestación supera el 86 % con tan solo 60 órdenes diarias, y su ocurrencia produce un error no controlado con posible exposición de información sensible. La corrección propuesta —basada en random_bytes() con 16 millones de combinaciones posibles por día— resuelve el problema con un esfuerzo mínimo de 5 a 8 horas-persona.

Las tres acciones prioritarias propuestas en la Actividad 1 (implementación de suite de pruebas automatizadas, refactorización del manejo de errores y ejecución de pruebas de carga con encuesta de satisfacción) conforman una hoja de ruta coherente y viable para elevar el nivel de calidad del sistema SGMOT a los estándares requeridos por un proyecto de software de nivel productivo.
